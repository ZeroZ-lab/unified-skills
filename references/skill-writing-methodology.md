# Skill Writing Methodology — 概念锚点式压缩法

> 技能编写的核心原则：用经典概念名压缩解释，用本地执行规则保证落地。

## 核心思想

Agent 执行技能时，不需要"理解为什么"——它需要的是"在什么条件下做什么"。一段解释 HTTP 语义的段落，和一句 `RFC 9110` 加 3 行执行规则，在 Agent 行为上等价，但 token 成本差 5-10 倍。

**概念锚点 = 权威名 + 本地执行规则。** 权威名（RFC、设计模式名、书籍名、行业惯例名）负责指代整个知识体系；本地执行规则（2-5 行具体指令）负责把通用知识映射到当前技能的执行上下文。两者缺一不可：只有权威名是空泛建议，只有执行规则是重复造轮子。

## 技能分类与目标规格

| 类型 | 特征 | 目标行数 | 概念锚点密度 |
|------|------|---------|-------------|
| 路由型 | 判断入口、分发 persona、组装调用链 | 80-150 | 低——流程本身即规则 |
| 设计型 | 证据收集、创作决策、多方案对比 | 200-350 | 中——锚点用于约束创作自由度 |
| 执行型 | Task-by-task 实现、TDD 循环、增量生成 | 150-300 | 高——锚点替代解释性步骤 |
| 治理型 | 审查门、合规检查、发布审计 | 120-250 | 高——锚点替代标准引用 |

**行数 = 主 SKILL.md 不含 frontmatter 的行数。** 辅助文件（长模板、评分表、示例库）不计入。

## 压缩操作手册

### 操作 1：识别可压缩段落

以下信号表明一段内容可以用概念锚点压缩：

- **解释性段落**：用 5+ 行解释一个行业已有标准名称的概念（如用段落解释幂等性 → `Idempotency-Key`）
- **教程式步骤**：逐步教导一个已有方法论的做法（如用 10 行教 Contract-first → 直接声明 `Contract-first / Schema-first`）
- **原则列举**：列举通用原则而非本技能特有规则（如"错误响应应该一致" → `RFC 9457 Problem Details`）
- **背景铺垫**：在进入流程前先解释"为什么这个阶段重要"（直接删除，用 Iron Law 或 HARD-GATE 代替）

### 操作 2：选择锚点并补充执行规则

锚点来源优先级：

1. **RFC / 规范号**（如 RFC 9110、RFC 9457、WCAG 2.1 AA）—— 互联网/行业标准的最高锚点
2. **设计模式名**（如 Strategy Pattern、Observer、CQRS）—— 软件工程通用知识
3. **书籍/论文名**（如 Hyrum's Law、Clean Architecture、DDD）—— 含义精确、争议少的经典
4. **行业惯例名**（如 Contract-first、Schema-first、Shift-Left）—— 需要补充本地约束的

执行规则模板：

```markdown
### [锚点名]

[一句话说明锚点在本技能中的含义]

**执行规则：**
- [具体指令 1]
- [具体指令 2]
- [具体指令 3]
```

执行规则必须满足：
- 每条是祈使句（"做 X"、"返回 409"、"禁止 Y"）
- 每条可检测（能用 grep/测试/代码审查验证）
- 总共 2-5 行，不超过 5 行（超过说明锚点选错了或规则太细碎）

### 操作 3：不可压缩的内容

以下内容必须原样保留在 SKILL.md 中，不得用锚点替代：

- **红旗 STOP 清单**：每条必须是可检测条件 + STOP 动作
- **验证失败处理表**：每行是具体失败场景 → 具体处理动作
- **说辞表**：三列表格，后果列必须量化
- **输出模板**：本技能产出的具体格式
- **验证清单**：本技能完成时的 checklist
- **Agent Dispatch Contract**：本技能的 persona 调用规则
- **入口/出口**：前置加载和输出路径
- **Iron Law / HARD-GATE**：不可绕过的硬纪律

这些是技能的行为骨架——它们定义"在什么条件下做什么"，锚点压缩的只是"为什么这样做"的解释性背景。

### 操作 4：该移入 references/ 的内容

- 设计模式详解（超过 5 行的模式说明）
- 外部标准全文引用（如完整的 HTTP 状态码表）
- 行业案例库（如"73 个真实网站设计系统"）
- 评分标准表（如五轴评分细则）
- 示例集合（如好/坏示例超过 3 对时）

移入后在 SKILL.md 中保留一行引用：`详见 references/xxx.md`。

## 写作纪律

### 每个技能必须包含

```
## 入口/出口
## 何时不使用
## 流程（或核心锚点）
## 常见说辞（三列）
## 红旗 — STOP
## 验证失败处理
## 输出模板
## 验证清单
```

### 强纪律技能额外包含

```
## Iron Law（含 HARD-GATE 标签）
```

### 每个 Step 必须满足

- 标题含动词
- 有 Checkpoint 或与验证清单的检查项对应
- 不超过 5 行解释（超过 = 该提取为辅助文件）

## 压缩质量检查

压缩完成后用以下标准自检：

| 维度 | 合格 | 不合格 |
|------|------|--------|
| 锚点密度 | 每 50 行至少 1 个锚点 | 大段纯散文无锚点 |
| 执行规则可操作性 | 每条是祈使句、可检测 | 含"应该"、"建议"、"可以考虑" |
| 行为骨架完整 | 8 个必要章节全部存在 | 缺少任意必要章节 |
| 无重复 | 锚点不重复解释同一概念 | 两个锚点或两段散文讲同一件事 |
| references/ 对齐 | SKILL.md 引用了存在的文件 | 引用了不存在的文件 |
| token 效率 | 删除解释段落后 Agent 行为不变 | 压缩后执行行为退化 |

## 压缩流程

1. **读完全文**——理解每个章节为什么存在
2. **标记可压缩段**——用操作 1 的信号扫描
3. **选择锚点**——用操作 2 的优先级匹配
4. **写执行规则**——每锚点 2-5 行祈使句
5. **保留行为骨架**——用操作 3 的清单验证
6. **提取长内容**——用操作 4 移入 references/
7. **跑 `./validate`**——确保合同一致
8. **行为等价测试**——压缩前后对同一指令的执行结果一致

## 反模式

- **过度压缩**：把本地规则也压缩成锚点，导致失去执行特异性。锚点压缩的是背景知识，不是执行规则。
- **锚点堆砌**：列了 10 个锚点但没有执行规则。没有执行规则的锚点是空泛建议。
- **格式漂移**：压缩时改变了说辞表格式、输出模板格式或验证清单格式。这些格式是 `./validate` 的检查目标。
- **删除红线**：为了压缩删除了红旗 STOP、HARD-GATE 或 Iron Law。这些是不可压缩的行为硬门。
