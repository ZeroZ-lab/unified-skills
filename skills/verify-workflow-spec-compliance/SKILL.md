# Spec Compliance — 功能完整性审查


## 入口/出口
- **入口**: 已完成的功能代码、即将合并的 PR、对应的 spec 文档
- **出口**: spec 合规性审查报告（通过/不通过 + 遗漏清单）
- **指向**: 通过 → `verify-quality-code-quality`；不通过 → 退回 build 补齐功能
- **假设已加载**: CANON.md

## 何时不使用
- 没有对应 spec 文档的代码变更（此时无法验证功能完整性）
- 代码还在开发中，未声称完成（此时不应进入审查阶段）

## Iron Law

<HARD-GATE>
spec 中的每个需求必须在代码中找到对应实现。
spec 中的每个验收标准必须有对应测试。
未实现的需求 = 不完整的功能 = 不能通过审查。
实现了 spec 之外的功能 = scope creep = 需要解释或退回。
</HARD-GATE>

### Step 1: 定位 spec 文档

找到对应的 spec 文档：
- 标准路径：`docs/features/YYYYMMDD-<name>/01-spec.md`
- Bug 修复：`docs/bugs/<name>/01-root-cause.md` + `02-fix-plan.md`
- 如果找不到 spec，询问用户 spec 位置或要求先创建 spec

### Step 2: 提取 spec 需求清单

逐行读取 spec，提取所有功能需求：

**需求类型识别：**
- **功能需求** — "系统应该..."、"用户可以..."、"支持..."
- **边界条件** — "当...时"、"如果...则"、"处理...情况"
- **错误场景** — "失败时..."、"无效输入..."、"异常..."
- **验收标准** — "验证..."、"确保..."、"检查..."

**提取格式：**
```markdown
## Spec 需求清单

### 功能需求
- [ ] [spec:L12] 用户可以创建新任务
- [ ] [spec:L34] 任务支持标题、描述、截止日期三个字段
- [ ] [spec:L56] 任务列表按创建时间倒序排列

### 边界条件
- [ ] [spec:L78] 标题为空时显示错误提示
- [ ] [spec:L90] 截止日期早于当前时间时警告用户

### 错误场景
- [ ] [spec:L102] 网络请求失败时显示重试按钮
- [ ] [spec:L124] 数据库连接失败时记录错误日志

### 验收标准
- [ ] [spec:L146] 创建任务后列表自动刷新
- [ ] [spec:L168] 所有操作在 200ms 内响应
```

### Step 3: 逐项验证实现

对每个需求，在代码中查找对应实现：

**验证方法：**
1. **功能需求** — 搜索相关函数、组件、API 端点
2. **边界条件** — 检查 if/else、switch、guard clause
3. **错误场景** — 检查 try/catch、错误处理逻辑
4. **验收标准** — 检查测试文件中的对应测试

**标记结果：**
```markdown
- [x] [spec:L12] 用户可以创建新任务
  → 实现: `src/tasks/TaskCreator.tsx:45-67`
  → 测试: `tests/tasks/TaskCreator.test.tsx:12-24`

- [ ] [spec:L34] 任务支持标题、描述、截止日期三个字段
  → 实现: `src/tasks/Task.ts:8-12` ⚠️ 缺少截止日期字段
  → 测试: 无对应测试

- [x] [spec:L56] 任务列表按创建时间倒序排列
  → 实现: `src/tasks/TaskList.tsx:89-91`
  → 测试: `tests/tasks/TaskList.test.tsx:34-42`
```

### Step 4: 识别 Scope Creep

检查代码中是否有 spec 未提及的功能：

**检查范围：**
- 新增的 API 端点
- 新增的 UI 组件
- 新增的数据库表或字段
- 新增的配置选项

**判断标准：**
- **合理扩展** — 实现 spec 需求的必要支撑（如内部工具函数）
- **Scope Creep** — spec 未要求的用户可见功能

**标记方式：**
```markdown
## Scope Creep 检查

- ⚠️ `src/tasks/TaskFilter.tsx` — spec 未提及任务过滤功能
  → 需要解释：这是必要的吗？还是应该移除？

- ✅ `src/tasks/utils/validateTask.ts` — 内部验证工具，支持 spec:L78 边界条件
  → 合理扩展
```

### Step 5: 生成合规性报告

汇总审查结果：

```markdown
# Spec Compliance 审查报告

**Spec 文档:** `docs/features/YYYYMMDD-<name>/01-spec.md`
**审查时间:** YYYY-MM-DD HH:MM
**审查结果:** 通过 / 不通过

## 需求覆盖率

- 功能需求: 8/10 (80%)
- 边界条件: 4/5 (80%)
- 错误场景: 2/3 (67%)
- 验收标准: 6/8 (75%)

**总体覆盖率: 20/26 (77%)**

## 遗漏需求（Critical）

以下需求在 spec 中明确要求，但代码中未实现：

1. **[spec:L34] 任务支持截止日期字段**
   - 当前状态: Task 类型只有 title 和 description
   - 影响: 用户无法设置任务截止日期
   - 建议: 在 `src/tasks/Task.ts` 添加 `dueDate?: Date` 字段

2. **[spec:L90] 截止日期早于当前时间时警告用户**
   - 当前状态: 无对应验证逻辑
   - 影响: 用户可能设置过期的截止日期而不自知
   - 建议: 在 `src/tasks/TaskCreator.tsx` 添加日期验证

3. **[spec:L102] 网络请求失败时显示重试按钮**
   - 当前状态: 只显示错误消息，无重试机制
   - 影响: 用户遇到临时网络问题时需要刷新页面
   - 建议: 在 `src/tasks/TaskList.tsx` 添加重试按钮

## Scope Creep（需要解释）

以下功能在代码中实现，但 spec 未提及：

1. **任务过滤功能** (`src/tasks/TaskFilter.tsx`)
   - 允许用户按状态、标签过滤任务
   - 问题: spec 未要求此功能，是否必要？

## 测试覆盖缺口

以下验收标准缺少对应测试：

- [spec:L34] 任务支持三个字段 — 无测试
- [spec:L90] 截止日期验证 — 无测试
- [spec:L168] 200ms 响应时间 — 无性能测试

## 审查结论

**不通过** — 3 个 Critical 遗漏需求必须在合并前实现。

**下一步:**
1. 补齐遗漏的 3 个功能需求
2. 为所有验收标准添加测试
3. 解释或移除 Scope Creep 功能
4. 重新提交审查
```

### Step 6: 判定通过/不通过

**通过标准:**
- 所有功能需求都有对应实现（100%）
- 所有边界条件都有对应处理（100%）
- 所有错误场景都有对应路径（100%）
- 所有验收标准都有对应测试（≥90%）
- 无未解释的 Scope Creep

**不通过标准:**
- 任何功能需求缺失 → Critical，必须修复
- 任何边界条件缺失 → Critical，必须修复
- 任何错误场景缺失 → Important，强烈建议修复
- 验收标准测试覆盖 <90% → Important，强烈建议补齐
- 有未解释的 Scope Creep → 需要解释或移除

**如果不通过:**
- 退回 build 阶段
- 提供清晰的遗漏清单和修复建议
- 修复后重新提交 spec compliance 审查
- 不进入 code quality 审查阶段

## 审查标准

**批准标准:** spec 的每个需求都在代码中找到对应实现，且有测试验证。

**拒绝标准:** 任何功能需求或边界条件缺失。

**灰色地带处理:**
- spec 描述模糊 → 询问用户澄清 spec 意图
- 实现方式与 spec 描述不完全一致但达到同样效果 → 标记为"实现方式差异"，由用户判断
- spec 要求的功能在技术上不可行 → 标记为"技术约束"，建议修改 spec

## 常见说辞

| 说辞 | 现实 | 后果 |
|------|------|------|
| "功能基本实现了，细节之后补" | "基本实现"意味着不完整。spec 是契约，不是建议。 | 不完整的功能合入主干后，"之后补"平均 = 永不。用户遇到缺失功能时，团队需要紧急修复，成本 10x。 |
| "这个边界情况很少见，可以不处理" | 边界情况是 spec 明确要求的，不是可选项。 | 生产环境中，"很少见"的边界情况会在最糟糕的时刻出现（如客户演示、重要发布）。未处理的边界情况 = 潜在的线上故障。 |
| "测试可以之后补，先合代码" | 没有测试 = 无法验证功能正确性。 | 无测试的代码在后续重构时容易引入回归 bug。"之后补测试"的代码，90% 永远不会有测试。 |
| "这个功能虽然 spec 没写，但用户肯定需要" | spec 是需求的唯一来源。未经批准的功能 = scope creep。 | 未经讨论的功能可能与产品方向冲突，或引入不必要的复杂度。合入后再移除的成本远高于合入前讨论。 |

**违反字面规则就是违反精神。** 没有灰色地带。

## 红旗

<HARD-GATE>
以下任何一个出现，立即标记为不通过：

- spec 的功能需求在代码中找不到对应实现
- spec 的边界条件在代码中没有处理逻辑
- spec 的验收标准没有对应测试
- 代码实现了 spec 未提及的用户可见功能，且无解释
- 审查者声称"功能基本实现"但实际有遗漏
- 审查者跳过某些需求，声称"不重要"或"之后补"
</HARD-GATE>

## 验证清单

- [ ] 已定位 spec 文档
- [ ] 已提取所有功能需求、边界条件、错误场景、验收标准
- [ ] 已逐项验证代码实现
- [ ] 已检查 scope creep
- [ ] 已检查测试覆盖
- [ ] 已生成合规性报告
- [ ] 已判定通过/不通过
- [ ] 如果不通过，已提供清晰的遗漏清单和修复建议

## 与 Code Quality 审查的边界

**Spec Compliance 关注"实现了什么":**
- spec 的需求是否都实现了？
- 实现的功能是否符合 spec 描述？
- 有没有实现 spec 之外的功能？

**Code Quality 关注"如何实现":**
- 实现的代码质量如何？（五轴：Correctness/Readability/Architecture/Security/Performance）
- 代码是否遵循最佳实践？
- 代码是否易于维护？

**分界线:**
- "这个函数没有处理 null 输入" → 如果 spec 要求处理 null，属于 Spec Compliance；如果 spec 未提及，属于 Code Quality（Correctness 轴）
- "这个变量命名不清晰" → Code Quality（Readability 轴）
- "这个功能缺少错误处理" → 如果 spec 要求错误处理，属于 Spec Compliance；如果 spec 未提及，属于 Code Quality（Correctness 轴）

**原则:** spec 明确要求的 = Spec Compliance；spec 未提及但应该有的 = Code Quality。
