# Content Writing — 内容写作


## 入口/出口
- **入口**: spec 的 `artifact_type` 为 `document` / `article` / `deck`，或当前切片需要写作、编辑、叙事结构
- **出口**: 符合受众、目标、事实和语气要求的内容草稿或定稿
- **指向**: 需要版式 → `build-content-layout`；完成后 → `verify-content-review`
- **假设已加载**: CANON.md + `define-workflow-spec`

## 何时不使用
- 纯软件实现，不涉及面向读者的内容
- 只做视觉构图且没有文案变化
- 用户明确要求保留原文不改，只做格式化

## Iron Law

内容不是字数堆叠。每一段都必须服务一个明确读者目标；无法说明作用的段落应删除或重写。

## 流程

### Step 1：确认内容契约

从 spec 读取：
- `artifact_type`
- 目标读者或观众
- 交付格式（文章、报告、PPT 文案、讲稿、页面文案）
- 成功标准
- 不做清单和语气边界

如果缺少目标读者，先按 CANON 第 9 条询问。不要为“所有人”写内容。

### Step 2：建立内容骨架

先写结构，再写正文：

```
目标读者 → 核心结论 → 支撑论点 → 证据/例子 → 行动或结尾
```

按产物类型调整：
- `document`: 摘要、背景、分析、建议、附录
- `article`: 标题、开头、核心论点、例子、结尾
- `deck`: 一页一个消息，标题先讲结论，再放支撑内容

### Step 3：生成最小可审稿版本

先完成一个可读版本，不追求最终措辞：
- 每节有明确目的
- 每段只有一个主张
- 关键事实有来源或标注为待验证假设
- 标题能单独串成完整逻辑

### Step 4：编辑轮次

按顺序编辑，不能混在一起：
1. **结构编辑** — 顺序是否合理，是否缺关键段落
2. **逻辑编辑** — 论点和证据是否匹配
3. **语言编辑** — 语气、节奏、简洁度
4. **格式编辑** — 标题层级、列表、引用、注释

### Step 5：留下验证证据

记录：
- 内容版本路径
- 已核查事实列表
- 未核查但保留的假设
- 主要编辑决策

## Artifact Type 指南

| artifact_type | 写作重点 | 不该做 |
|---------------|---------|--------|
| `document` | 完整、可引用、结构清楚 | 像演讲稿一样只写短句 |
| `article` | 观点鲜明、节奏顺畅、读者愿意读完 | 堆砌目录式小标题 |
| `deck` | 一页一个结论、标题串成故事 | 把文章复制进幻灯片 |

## 与其他技能配合

- 需要版式、页面布局、视觉层级 → `build-content-layout`
- 需要事实、逻辑、语气审查 → `verify-content-review`
- 需要导出最终文件 → `ship-artifact-export`

## 验证失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| 读者不明确 | 停止写作，询问目标读者和使用场景 |
| 内容没有核心结论 | 回到骨架，先写一句话结论 |
| 事实无法验证 | 标注为假设、替换为可验证事实，或询问是否保留 |
| 语气不一致 | 选定一个语气标准，统一改写 |

## 常见说辞

| 说辞 | 现实 |
|------|------|
| "先写多一点再删" | 没有结构的长文只会增加编辑成本。先骨架，再正文。 |
| "PPT 文案就是文章拆页" | PPT 是演示，不是分页文章。一页只能承担一个消息。 |
| "这个事实大概对" | 不确定的事实会污染整篇内容。验证或标注假设。 |
| "语气不重要" | 语气决定读者是否相信你为他写，而不是随便生成。 |

## 红旗 — STOP

- 没有目标读者就开始写
- 标题无法串成完整逻辑
- 一段里有多个互相竞争的主张
- 把未验证事实写成确定结论
- deck 页面文案超过演示者能自然讲完的量
- article/document 没有明确结论或下一步

## 验证清单

- [ ] artifact_type 已读取
- [ ] 目标读者和使用场景明确
- [ ] 内容骨架先于正文建立
- [ ] 关键事实已验证或标注假设
- [ ] 标题层级和逻辑顺序清楚
- [ ] 语气与 spec 一致
- [ ] 已记录内容路径和验证证据
