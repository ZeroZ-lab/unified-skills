> 用于模式 C（Subagent 两阶段审查流水线）中分派实现任务。主 agent 将此模板填充后交给 subagent。

**涉及文件:** {{FILES}}

---

## 前置自检

开始任何实现之前，逐条回答以下问题。有任何一条无法回答 → 状态设为 `NEEDS_CONTEXT`，列出缺失项。

1. 我是否理解任务要交付的最终产物是什么？
2. 我是否知道产物的验收标准（acceptance criteria）？
3. 我是否清楚哪些文件需要修改、哪些文件不能碰？
4. 我是否掌握了所有必要的上下文（spec、依赖关系、约束条件）？
5. 我是否知道测试必须覆盖哪些场景？

## 产物类型适配

根据 `{{ARTIFACT_TYPE}}` 选择对应的工作方式：

### software（软件）

遵循 TDD 循环：
1. 写失败测试 → 2. 最小实现 → 3. 测试通过 → 4. 重构
- 每个循环 < 50 行变更
- 没有测试先失败的代码 = 不存在的代码（CANON 第 4 条）

### document（文档）

1. 事实核查 — 每个断言必须有来源
2. 逻辑流 — 段落之间有因果递进，不是列表堆砌
3. 完整性 — 对照 spec 检查是否有遗漏章节

### deck（演示文稿）

1. 叙事弧 — 开头设问题、中间给方案、结尾给行动
2. 信息密度 — 每页一个核心观点，不超过 30 字标题 + 3 个要点
3. 视觉一致性 — 统一字体/色彩/间距

### visual（视觉产物）

1. 层级 — 通过大小/色彩/位置建立信息优先级
2. 对齐 — 网格系统，元素不随意放置
3. 可访问性 — 色彩对比度、文字可读性

---

## 自审清单

实现完成后，逐条检查。有任何一条不通过 → 修正后再报告。

- [ ] 实现是否完全匹配 spec 要求（无遗漏、无多余）？
- [ ] 边界情况是否已处理（空值、异常、并发、极端输入）？
- [ ] 测试是否已编写并通过（software 类型必须有）？
- [ ] 变更是否限制在 `{{FILES}}` 范围内，未触碰无关文件？
- [ ] 是否遵循了 CANON 第 3 条（只改该改的）？

---

## 升级协议

完成后必须输出以下格式的报告。选择最匹配的状态：

| 状态 | 含义 |
|------|------|
| **DONE** | 任务完成，所有自审清单通过，测试通过 |
| **DONE_WITH_CONCERNS** | 任务完成但有保留意见，需主 agent 知晓 |
| **BLOCKED** | 无法继续，需要人类介入 |
| **NEEDS_CONTEXT** | 缺少必要信息，无法完成，列出缺失项 |

---

## 输出格式

```
状态: <DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT>

摘要: <一段话描述做了什么>

变更文件:
  - <文件路径>: <变更摘要>
  - ...

关切: <仅 DONE_WITH_CONCERNS 时填写，列出保留意见>

升级: <仅 BLOCKED 或 NEEDS_CONTEXT 时填写，说明原因或缺失信息>
```
