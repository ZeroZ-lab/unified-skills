# Receiving Review — 接收审查反馈


## 入口/出口
- **入口**: 审查反馈（来自 human partner 或外部 reviewer）
- **出口**: 逐项处理后的反馈响应
- **指向**: 处理完成后回到 `/build` 修复
- **假设已加载**: CANON.md

## Iron Law

<HARD-GATE>
收到反馈不等于同意反馈。每条反馈必须经过独立的技术评估——盲目同意和盲目拒绝一样危险。
</HARD-GATE>

## 流程

### Step 1: READ — 完整阅读

完整阅读所有反馈，不做任何反应。不要边读边改。

- 通读全部反馈，建立整体印象
- 记录反馈条目数量
- 不开始任何代码修改
- 如果反馈很长，先按编号整理

### Step 2: UNDERSTAND — 理解意图

用自己的话重述每条反馈的要求。不理解的先问。

- 对每条反馈，用自己的话复述：这条反馈要我改什么？为什么要改？
- 复述不出来 = 没理解 → 提问澄清
- 每条不明确的反馈都单独提问，不等"攒够了再问"

### Step 3: VERIFY — 验证事实基础

对照代码库验证反馈的事实基础。反馈说"这里有问题"→ 代码里真的有问题吗？

- 打开对应代码，确认反馈指向的位置存在
- 确认反馈描述的现象是否真实存在
- 确认反馈引用的上下文是否完整（有没有遗漏的前提条件）
- 事实不成立 → 记录证据，准备技术性回应

### Step 4: EVALUATE — 评估合理性

这条反馈对这个代码库技术上合理吗？执行上下文和约束。

- 对照代码库的已有模式评估
- 执行约束：性能要求、兼容性、迁移成本
- 评估实施风险和副作用
- 给出判断：采纳 / 反驳 / 部分采纳

### Step 5: RESPOND — 技术性回应

技术性确认或理由充分的反驳。见下方反驳指南。

- 采纳：说明理解和技术依据，不是空洞附和
- 反驳：见下方反驳指南
- 每条反馈一条回应，不混在一起

### Step 6: IMPLEMENT — 逐条实施

一次处理一条，每条处理后跑测试。

- 按实施顺序排列（见下方）
- 每条处理完后跑测试
- 测试失败 → 回退该条修改，重新评估

## 来源区分

### trusted human partner
- 理解后直接实施，信任其意图
- 仍需 Step 1-2（完整阅读 + 理解）
- 不理解的部分仍然要提问

### 外部 reviewer
- 需要通过 5 点验证清单：

```
外部反馈验证清单：
1. 事实准确？ — 反馈描述的现象在代码中真实存在吗？
2. 上下文完整？ — 反馈有没有遗漏关键前提条件？
3. 约束执行？ — 反馈的必须是否执行了项目的现实约束？
4. YAGNI 检查？ — 必须的改动是否是当前真实需要的？
5. 实施成本合理？ — 收益是否大于实施成本？
```

## YAGNI 检查

reviewer 建议"properly implement"或"add abstraction"时，先确认真实使用场景：

```bash
# grep 代码库确认真实使用场景
grep -r "<模式/函数/抽象名>" --include="*.ts" --include="*.tsx"
```

- 1 个使用场景 = 不抽象，保持内联
- 2 个使用场景 = 可抽象可保持，看代码库风格
- 3+ 个使用场景 = 抽象合理
- 没有第三个使用场景 = 不抽象

## 反驳指南

### 何时反驳
- 有技术证据（代码、测试结果、性能数据）
- 有量化影响（延迟增加、包体积膨胀、复杂度上升）
- 违反代码库已有模式且无充分理由

### 如何反驳
格式：事实 + 数据 + 替代方案

```
反驳模板：
"这个必须会增加 ~200ms 延迟（基准 X ms），
替代方案是 Y，
因为 [技术理由]。"
```

### 如何纠正自己的反驳
如果你发现反驳是错的：
- 直接说："我之前的反驳不成立，因为..."
- 不找借口，不淡化
- 立即切换到实施模式

## 禁止行为（红旗区）

<HARD-GATE>
以下回应模式严格禁止——它们是空洞的同意，不是技术性回应：

| 禁止说辞 | 为什么禁止 |
|---------|-----------|
| "You're absolutely right!" | 空洞的同意，没有技术分析 |
| "Great point!" | 没有技术分析的附和 |
| "Thanks for catching that!" | 感谢不是技术回应 |
| "I'll fix that right away" | 没有评估就承诺修复 |
| 对所有反馈说"好的" | yes-machine 模式 |
| 沉默接受所有建议 | 放弃独立技术判断 |
</HARD-GATE>

## 实施顺序

```
1. 澄清不清楚项 FIRST   ← 不理解的先问，不猜
2. 阻塞性问题            ← Critical 级别
3. 简单修复              ← 快速处理的 Nit
4. 复杂修复              ← 需要设计或重构的项
```

## 常见说辞

| 说辞 | 现实 | 后果 |
|------|------|------|
| "reviewer 总是对的" | 盲目信任和盲目拒绝一样危险。每条反馈需要独立技术评估。 | 盲目接受的反馈中有 ~20% 在项目上下文中不适用——执行后引入新问题或违反已有模式，返工时间 > 评估时间。 |
| "不能反驳 reviewer" | 技术讨论是工程文化，不是服从训练。有证据的反驳是贡献。 | 压制异议 = 团队停止思考。没有技术辩论的审查退化为人情盖章，代码质量持续下降。 |
| "先全部改完再说" | 每条反馈是独立决策，逐条评估。批量接受 = 放弃判断。 | 批量修改后无法定位哪条反馈引入了新 bug。回退范围 = 全部改动，而不是单条有问题的修改。 |
| "反馈太多了，挑着改" | 每条都读，每条都评估。评估后决定不改，但不能不读。 | 未读的反馈中可能包含 Critical 问题——安全漏洞、数据丢失风险。跳过 = 对未知风险视而不见。 |
| " reviewer 比我懂" | reviewer 看到的是 snapshot，你活在代码库里。上下文不对称。 | reviewer 的建议可能基于过时假设或遗漏了局部约束。盲目执行 = 用 snapshot 判断覆盖 lived experience，引入与现有代码矛盾的设计。 |

**违反字面规则就是违反精神。** 没有灰色地带。

## 红旗

<HARD-GATE>
以下任何一个出现，立即停止：

- 未读完全部反馈就开始改代码
- 对所有反馈说"好的"（yes-machine 模式）
- 反驳时没有技术证据（纯主观偏好）
- 混淆"我不同意"和"你是错的"
- 一条反馈失败后继续改下一条（不回退）
- 跳过测试直接处理下一条反馈
- 把"感谢"当作对反馈的技术回应
- 不提问就直接猜测反馈意图
</HARD-GATE>

- [ ] 实施顺序正确（澄清 → 阻塞 → 简单 → 复杂）
- [ ] 每条实施后测试通过
- [ ] 响应记录在 `docs/features/<name>/review.md`
