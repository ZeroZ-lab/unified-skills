# Content Layout — 版式与视觉层级


## 入口/出口
- **入口**: 内容草稿已存在，或 spec 的 `artifact_type` 为 `document` / `deck` / `visual`
- **出口**: 可读、稳定、符合交付格式的版式或视觉稿
- **指向**: 内容问题 → `build-content-writing`；视觉审查 → `verify-visual-review`
- **假设已加载**: CANON.md

## 何时不使用
- 只修改纯文本内容，不涉及排版或视觉呈现
- 纯软件后端功能，无用户可见界面或交付物
- 用户提供了不可修改的模板且只允许填充内容

## Iron Law

版式服务理解。任何装饰、布局或视觉层级如果不能帮助读者更快理解，就应该删除或降低权重。

## 流程

### Step 1：读取版式约束

从 spec 或模板确认：
- 交付格式：DOCX、PDF、PPTX、PNG、SVG、网页截图等
- 画布尺寸、页边距、字体、品牌色、图片规格
- 使用场景：阅读、汇报、打印、投屏、社媒、交付归档
- 必须保留的模板元素

### Step 2：建立信息层级

先排序内容重要性：
1. 读者必须第一眼看到的内容
2. 支撑理解的内容
3. 可延后阅读的细节
4. 可移到附录、备注或删掉的内容

### Step 3：布局

按产物类型处理：
- `document`: 标题层级、页边距、段落间距、图表说明、页眉页脚
- `deck`: 一页一个中心消息，控制每页信息密度，保留 speaker notes 空间
- `visual`: 构图、对齐、留白、对比度、导出尺寸

### Step 4：稳定格式

检查动态内容不会破坏版式：
- 长标题换行后不遮挡正文
- 图片比例不失真
- 表格或图表在页面内完整显示
- 页码、注脚、备注不覆盖主体内容

### Step 5：导出前自检

记录：
- 源文件路径
- 预览或截图路径
- 目标导出格式
- 已知限制

## Artifact Type 指南

| artifact_type | 版式重点 | 验证方式 |
|---------------|---------|---------|
| `document` | 可读性、层级、打印/导出稳定 | 翻页检查、PDF 预览 |
| `deck` | 叙事节奏、页面密度、投屏可读 | 全屏预览、逐页检查 |
| `visual` | 构图、对齐、对比、规格 | 实际尺寸预览、导出检查 |

## 验证失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| 内容太多放不下 | 回到 `build-content-writing` 压缩或拆分 |
| 页面视觉重点混乱 | 降低次要元素权重，只保留一个主焦点 |
| 导出后错位 | 调整固定尺寸、图片比例和字体策略后重新导出 |
| 品牌约束缺失 | 使用保守默认样式，并记录假设 |

## 常见说辞

| 说辞 | 现实 |
|------|------|
| "内容对了，丑一点没关系" | 版式差会降低理解效率，也会削弱可信度。 |
| "PPT 多放点信息显得充实" | 信息密度过高等于没有重点。 |
| "视觉稿看起来差不多" | 差不多通常意味着没有对齐、间距和层级标准。 |
| "导出最后再看" | 导出问题越晚发现，返工越大。 |

## 红旗 — STOP

- 没有明确画布或导出格式
- 一页/一屏有多个主焦点
- 文本靠缩小字号硬塞
- 图片被拉伸或裁切掉关键信息
- 对齐和间距没有一致规则
- 导出前没有实际预览

## 验证清单

- [ ] artifact_type 已读取
- [ ] 交付格式和尺寸明确
- [ ] 信息层级清楚
- [ ] 版式没有遮挡、溢出、错位
- [ ] 字体、颜色、间距一致
- [ ] 导出预览已检查
