# Content Review — 内容审查


## 入口/出口
- **入口**: document / article / deck 草稿或定稿已完成
- **出口**: 内容审查结论、阻塞问题、修订建议和验证证据
- **指向**: 有问题 → 回到 `build-content-writing`；通过 → `ship-artifact-export`
- **假设已加载**: CANON.md

## 何时不使用
- 纯视觉稿且没有文本或叙事
- 纯软件代码审查（使用 `verify-workflow-review`）
- 用户只要求格式转换，不要求内容判断

## Iron Law

没有读者目标、事实核查和逻辑链审查，就不能批准内容交付。

## 流程

### Step 1：确认审查对象

读取：
- `artifact_type`
- 目标读者
- 成功标准
- 内容文件路径
- 事实来源或引用材料

### Step 2：五轴内容审查

#### 1. Audience Fit（受众适配）
- 是否解决目标读者的问题？
- 前置知识假设是否合理？
- 用词是否符合读者身份和场景？

#### 2. Logic（逻辑）
- 结论是否明确？
- 论点顺序是否自然？
- 每个主张是否有支撑？
- 是否存在跳步、循环论证或前后矛盾？

#### 3. Accuracy（事实准确）
- 数字、日期、名称、引用是否可验证？
- 不确定信息是否标注为假设？
- 是否混淆事实、观点和建议？

#### 4. Voice（语气）
- 语气是否与 spec 一致？
- 是否过度营销、过度保守或不够直接？
- 是否有不必要的套话？

#### 5. Completeness（完整性）
- 是否覆盖验收标准？
- 是否缺少必要背景、例子、结论或下一步？
- 是否有多余内容干扰主线？

### Step 3：按严重性分类

| 前缀 | 含义 | 要求 |
|------|------|------|
| **Critical:** | 事实错误、核心逻辑错误、误导读者 | 必须修复 |
| 无前缀 | 影响理解或交付质量 | 合并/交付前处理 |
| **Consider:** | 可改善但不阻塞 | 可按时间取舍 |
| **Nit:** | 细节措辞 | 可选 |

### Step 4：记录审查证据

审查报告至少包含：
- 审查对象路径
- artifact_type
- 已核查事实清单
- Critical / 必须修复项
- 通过或退回结论

## Artifact Type 指南

| artifact_type | 审查重点 |
|---------------|---------|
| `document` | 完整性、引用、结构、可归档性 |
| `article` | 观点、节奏、读者吸引力、结论 |
| `deck` | 叙事线、页面标题、信息密度、讲述顺序 |

## 验证失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| 事实无法验证 | 标注为阻塞或改写为假设 |
| 逻辑链断裂 | 退回重排结构，不做局部措辞修补 |
| 受众不匹配 | 回到目标读者和使用场景重写 |
| deck 页面过载 | 拆页、删减或移入 speaker notes |

## 常见说辞

| 说辞 | 现实 |
|------|------|
| "读起来还行" | 审查需要证据，不是感觉。 |
| "这个事实大家都知道" | 交付物里的事实需要可验证或明确是常识假设。 |
| "先交付，之后再润色" | 内容交付后的误解成本通常比写作成本更高。 |
| "PPT 不需要逻辑那么严" | 演示更依赖逻辑，因为观众不能反复重读。 |

## 红旗 — STOP

- 没有目标读者
- 没有明确结论
- 关键事实无来源
- 标题串起来不成故事
- 内容和验收标准对不上
- 审查意见没有严重级别

## 验证清单

- [ ] artifact_type 已读取
- [ ] 目标读者和成功标准已对照
- [ ] 五轴内容审查完成
- [ ] 关键事实已核查或标注假设
- [ ] Critical 问题已列出
- [ ] 通过/退回结论明确
- [ ] 审查证据已记录
