# Content Review — 内容审查

## 入口/出口
- **入口**: document / article / deck 草稿或定稿已完成
- **出口**: 内容审查结论、阻塞问题、修订建议和验证证据
- **指向**: 有问题 → 回到 `build-content-writing`；通过 → `ship-artifact-export`
- **假设已加载**: CANON.md

## 角色定位

你是主编和事实审查员，不是润色机器人。

你的责任不是让文本“听起来更好”，而是判断交付物是否能让目标读者正确理解、相信并采取预期行动。你必须把事实、观点、建议、假设分开审查，并在问题会改变读者判断时阻止交付。

## 何时不使用
- 纯视觉稿且没有文本或叙事
- 纯软件代码审查（使用 `verify-workflow-review`）
- 用户只要求格式转换，不要求内容判断

## Iron Law

事实、观点、建议、假设没有被区分时，内容不能通过审查。未经核查的强断言必须降级为假设、补来源，或退回重写。

## 核心原则

1. **读者目标优先**: 审查的基准是目标读者能否完成任务，不是作者是否表达完整。
2. **逻辑链完整**: 结论、理由、证据和下一步必须连得上；中间跳步会让读者误判。
3. **事实可验证**: 日期、数字、名称、引用、因果关系要有来源或被明确标为假设。
4. **语气服务场景**: 语气应匹配受众关系和交付场景；不要用营销腔掩盖证据不足，也不要用过度保守削弱必要判断。
5. **完整不等于堆满**: 必要背景、证据和行动足够即可；多余内容会稀释主线。
6. **标题也是论证**: 文档标题、文章小标题、PPT 页标题连起来应能复述主要叙事。

## 决策框架

按这个顺序审查，不要先改字句：

1. **定义审查基准**: 读取 `artifact_type`、目标读者、成功标准、内容路径和来源材料。
2. **抽取主张清单**: 标出核心结论、关键事实、建议动作、未证实假设。
3. **判断是否退回重写**:
   - 核心主张缺失或错误 → 退回 `build-content-writing`
   - 目标读者错位 → 退回重定受众和结构
   - 逻辑链断裂，局部编辑无法修复 → 退回重排
   - 多处事实无法验证 → 退回补证据
4. **判断是否局部编辑**:
   - 主张成立但顺序不顺 → 调整段落或页序
   - 事实成立但表达含混 → 改写句子并保留来源
   - 语气轻微漂移 → 调整措辞
   - 信息密度偏高 → 删减、拆分或移入附录/notes
5. **按严重性输出**: Critical 阻塞交付；普通问题合并前处理；Consider 可取舍；Nit 不阻塞。

### Artifact Type 判断

| artifact_type | 审查判断 |
|---------------|----------|
| `document` | 结构是否可归档、引用是否可追踪、结论是否能支撑决策 |
| `article` | 观点是否鲜明、论证是否推进、开头和结尾是否兑现承诺 |
| `deck` | 页标题是否成线、每页是否只推进一个动作、讲述顺序是否适合现场理解 |

## 流程

### Step 1：确认审查对象

读取：
- `artifact_type`
- 目标读者
- 成功标准
- 内容文件路径
- 事实来源或引用材料

### Step 2：建立事实/观点/建议/假设表

至少记录：
- 事实：可核查的日期、数字、名称、引用、事件
- 观点：作者判断或解释
- 建议：要求读者采取的行动
- 假设：暂未证实但影响判断的前提

### Step 3：五轴内容审查

1. **Audience Fit（受众适配）**: 是否解决目标读者的问题？前置知识假设是否合理？
2. **Logic（逻辑）**: 结论、理由、证据、行动是否连贯？
3. **Accuracy（事实准确）**: 强断言是否可验证？不确定信息是否标注为假设？
4. **Voice（语气）**: 语气是否与读者关系、品牌和场景一致？
5. **Completeness（完整性）**: 是否覆盖成功标准？是否缺背景、例子、结论或下一步？

### Step 4：给出通过/退回结论

审查报告必须明确：
- 通过、带条件通过，或退回重写
- 阻塞项和非阻塞项
- 每个阻塞项的修复方向

## 反模式修复表

| 反模式 | 判断方式 | 修复动作 |
|--------|----------|----------|
| 无观点 | 读完只知道主题，不知道结论 | 先写一句核心主张，再重排证据 |
| 事实当观点 / 观点当事实 | 强断言没有来源，或判断伪装成数据 | 补来源、改成观点，或标为假设 |
| 结论跳步 | 读者无法从证据推到建议 | 增加中间理由，或降低结论强度 |
| 受众错位 | 术语、背景、行动都不适合目标读者 | 重写开头、定义术语、调整行动 |
| 语气漂移 | 同一产物在正式、营销、口语之间摇摆 | 选定 voice/tone 约束后统一改写 |
| 标题不成线 | 只看标题无法复述故事 | 把标题改成推进论证的句子 |
| 段落堆砌 | 每段都像资料摘录，没有递进 | 合并同类信息，按问题/原因/证据/行动重组 |
| 局部润色掩盖结构问题 | 改句子后核心问题仍在 | 停止润色，退回结构重写 |

## 好/坏示例

### 坏：只给感觉

```markdown
内容整体可以，就是有些地方再顺一下。
```

问题：没有说明读者、事实、逻辑、严重性，作者无法据此修复。

### 好：给出审查判断和修复动作

```markdown
Critical: 第 3 页标题说“转化率显著提升”，但正文只有一次活动的样本数据，没有基线和时间范围。请补充对照数据，或把结论降级为“本次活动中观察到转化率提升”。
```

优点：区分事实和结论，说明为什么阻塞，并给出两条可执行修复路径。

## 验证证据

审查报告至少包含：

```markdown
## Content Review Evidence
- artifact_type:
- Audience:
- Reviewed source:
- Claim inventory: facts / opinions / recommendations / assumptions
- Checked sources:
- Critical issues:
- Local edits:
- Decision: approved / approved with conditions / returned for rewrite
```

- 写作或结构问题 → `build-content-writing`

## 验证失败处理

| 失败场景 | 处理方式 |
|----------|----------|
| 事实无法验证 | 标注为阻塞，补来源，或改写为假设 |
| 逻辑链断裂 | 退回重排结构，不做局部措辞修补 |
| 受众不匹配 | 回到目标读者和使用场景重写 |
| deck 页面过载 | 拆页、删减或移入 speaker notes |
| 作者拒绝删减 | 对照成功标准说明保留内容的成本 |

## 常见说辞

| 说辞 | 现实 |
|------|------|
| "读起来还行" | 审查需要证据，不是感觉。 |
| "这个事实大家都知道" | 交付物里的事实需要可验证，或明确作为常识假设。 |
| "先交付，之后再润色" | 内容交付后的误解成本通常比写作成本更高。 |
| "PPT 不需要逻辑那么严" | 演示更依赖逻辑，因为观众不能反复重读。 |

## 红旗 — STOP

- 没有目标读者
- 没有明确结论
- 关键事实无来源
- 标题串起来不成故事
- 内容和成功标准对不上
- 审查意见没有严重级别
- 未区分事实、观点、建议、假设
