# Decision Record — 架构决策记录 (ADR)


## 入口/出口
- **入口**: build 中面临技术选型、方案取舍、架构决策
- **出口**: `docs/features/<name>/adr/<num>.md` — 可追溯的决策文件
- **指向**: 决策记录后继续 build
- **前置加载**: CANON.md
- **输出路径**: 决策记录后继续 `build-workflow-execute`

## 何时不使用
- 只是变量命名、局部文件拆分或框架惯例内的实现细节
- 方案没有真实取舍，只有一个显然符合现有模式的做法
- 已有 ADR 覆盖当前决策，只需要引用而不是新建

## 何时写 ADR

**必须写:** 选择技术方案（X vs Y）、改变架构模式、引入新依赖、选择数据存储方案、采用新通信模式、决定废弃某个系统

**不必写:** 选择变量名、文件拆分方式、单行实现细节、框架惯例（如 Next.js 的文件路由已经是框架决策）

## Iron Law

```
每个重要设计决策必须记录。
"代码即文档"说明 WHAT，"ADR 即文档"说明 WHY。
没有 WHY 的决策在复盘时等于没做决策。
```

## ADR 模板

```markdown
# ADR-<NNN>: <简短标题>

**日期:** YYYY-MM-DD
**状态:** 提议 | 接受 | 取代 | 废弃

## 背景

[为什么需要做这个决策？什么约束驱使我们？]

## 选项

### 选项 A: <名称>

- 优点:
- 缺点:
- 对现有系统的影响:

### 选项 B: <名称>

- 优点:
- 缺点:
- 对现有系统的影响:

## 决策

**选择: <选项>**

理由: [为什么选这个、不选其他。一句话解释清楚。]

## 后果

### 正面的
- [我们会得到什么好处]

### 负面的  
- [我们必须承担什么代价、哪些功能受影响]

### 被放弃的
- [选择 A 放弃了什么；选择 B 放弃了什么]

## 相关
- [相关的 ADR、Issue、PR、文档引用]
```

## 决策生命周期

```
提议 ──→ 接受 ──→ (可能) 取代 ──→ (可能) 废弃
  │                           │
  └── 讨论后拒绝              └── 新 ADR 引用旧 ADR
```

**不删除旧 ADR。** Git 有历史，ADR 也有历史。旧 ADR 被"取代"或"废弃"仍然留存——说明为什么当时做了那个决定、以及为什么后来改变了。

## 何时不算 ADR

不是所有选择都值得 ADR。阈值检查：
- 影响 > 1 个模块/服务的架构 → ADR
- 未来不可轻易改变的（数据模型、外部依赖、通信协议）→ ADR
- 只有一个人理解的隐式选择 → ADR
- 单文件实现细节 → 代码注释
- 框架的默认做法 → 不需要 ADR

## 示例

```markdown
# ADR-001: 选择 Prisma 作为 ORM

**日期:** 2026-04-24
**状态:** 接受

## 背景
项目需要管理 PostgreSQL 中的用户、任务和评论。需要类型安全的查询和自动化迁移。

## 选项

### 选项 A: Prisma
- 优点: 声明式 schema、自动迁移、TypeScript 类型自动生成、活跃社区（45k+ GitHub stars）
- 缺点: 复杂查询性能不如 raw SQL、嵌套写入有限制
- 对现有系统: 较小影响；替代手动 SQL 查询

### 选项 B: Drizzle ORM
- 优点: 更接近 SQL、零运行时、更小的包体积
- 缺点: 社区较小、迁移工具不如 Prisma 成熟
- 对现有系统: 不可行——团队已有 Prisma 经验

### 选项 C: TypeORM
- 优点: 成熟、装饰器模式与 NestJS 友好
- 缺点: 装饰器与纯函数趋势不符、active issue 积压较多
- 对现有系统: 需要改写所有现有查询

## 决策
选择 Prisma（选项 A）。团队现有经验、类型安全要求、和自动迁移是决定因素。Drizzle 更接近 SQL 但社区尚未到我们需要的成熟度。

## 后果
- 正面: 类型安全的自动生成类型、声明式 schema、内建迁移工具
- 负面: 复杂查询可能需要 `$queryRaw`、包体积较大
- 放弃: Drizzle 的更轻量设计和更接近 SQL 的 API
```

## 常见说辞

| 说辞 | 现实 | 后果 |
|------|------|------|
| "代码就是文档" | 代码显示 WHAT。6 个月后 WHY 只能在 ADR 里找到。 | 重做决策耗时 ×3；同一路径被反复讨论，浪费团队 2-4 小时/次 |
| "决策很明显不需要记录" | 对你明显≠对下一个接手的人明显。12 个月后的同事不会共享你的 context。 | 接手者误推翻已有决策 → 返工 1-2 周；重复踩同样的坑 |
| "写了没人读" | 被问"为什么选 X"时——发 ADR 链接。被问第 5 次时 ADR 的价值就出来了。 | 口头解释 ×5 次 × 15 分钟 = 75 分钟；一条 ADR 只需写 10 分钟 |
| "记录所有选择" | ADR 是精选。只有那些影响架构、不可轻易改变、或有多种合理方案的决策才需要。 | 过多 ADR → 信噪比下降 → 真正重要的决策淹没在噪声中 |

## 红旗 — STOP

- 选择了不同方案的项目依赖，没人知道为什么
- "我记得我们讨论过 X，但不记得为什么不选 Y"
- 引入新依赖——没有记录其他竞争方案为什么没选
- 代码走了一条和项目约定不同的路——没有 ADR 解释
- 有人提议替换决策时的方案——没有 ADR 可查"为什么当初选它"

## 验证失败处理

| 验证项 | 失败表现 | 处理方式 |
|--------|----------|---------|
| 缺少背景 | ADR 只写了"选 X"，没说为什么需要决策 | 补充背景：约束、驱动力、现状问题 |
| 选项不足 | 只评估了 1 个方案，没有对比 | 至少添加 1 个竞争方案并评估优缺点 |
| 决策理由模糊 | 理由是"大家都用"或"感觉不错" | 替换为具体证据：性能数据、团队经验、约束匹配 |
| 后果缺失 | 只列正面后果，不列负面 | 补充负面后果和被放弃选项的价值 |
| ADR 未编号保存 | ADR 写了但没放到 docs/features/<name>/adr/ | 按编号保存到正确路径，确保可引用 |

## 好坏示例

### Good: 有对比、有理由、有后果
```markdown
## 选项
### 选项 A: Prisma
- 优点: 声明式 schema、自动迁移、TypeScript 类型自动生成
- 缺点: 复杂查询性能不如 raw SQL
### 选项 B: Drizzle ORM
- 优点: 更接近 SQL、零运行时
- 缺点: 社区较小、迁移工具不如 Prisma 成熟
## 决策
选择 Prisma（选项 A）。团队现有经验、类型安全要求、和自动迁移是决定因素。
## 后果
- 负面: 复杂查询可能需要 $queryRaw、包体积较大
- 放弃: Drizzle 的更轻量设计
```

### Bad: 无对比、理由模糊、后果缺失
```markdown
## 决策
选择 Prisma，因为大家都用。
## 后果
(无)
```

## 输出模板

```
ADR 完成：

编号: ADR-<NNN>
标题: <简短标题>
状态: <提议/接受/取代/废弃>
路径: docs/features/<name>/adr/<num>.md

选项评估:
  - 选项 A: <名称> → 优点/缺点/影响
  - 选项 B: <名称> → 优点/缺点/影响
决策: <选择>
理由: <一句话解释>
后果:
  - 正面: <列出>
  - 负面: <列出>
  - 放弃: <列出>
```

## 验证清单

- [ ] ADR 包含背景（为什么需要决策）
- [ ] 至少 2 个选项被评估（有"放弃"才有"选择"）
- [ ] 每个选项有优点/缺点/影响
- [ ] 决策理由清晰（不是"因为大家都用"）
- [ ] 正面和负面后果都列出
- [ ] 被放弃的选项被明确列出
- [ ] ADR 文件编号并保存到 `docs/features/<name>/adr/`
