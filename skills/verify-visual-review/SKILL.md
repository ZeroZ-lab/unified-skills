# Visual Review — 视觉审查


## 入口/出口
- **入口**: 视觉稿、PPT 页面、文档版式或导出预览已完成
- **出口**: 视觉审查结论、阻塞问题、截图/预览证据
- **指向**: 有问题 → `build-content-layout`；通过 → `ship-artifact-export`
- **假设已加载**: CANON.md

## 何时不使用
- 纯文本文章且没有版式要求
- 纯后端或不可见软件变更
- 用户明确只要求内容审查，不判断视觉呈现

## Iron Law

视觉审查必须看实际预览。只读源文件或凭想象判断版式，不算验证。

## 流程

### Step 1：确认预览对象

读取：
- `artifact_type`
- 源文件路径
- 导出或预览路径
- 目标尺寸和使用场景
- 品牌或风格约束

### Step 2：五轴视觉审查

#### 1. Hierarchy（层级）
- 第一视觉焦点是否正确？
- 标题、正文、注释、图表的权重是否清楚？
- 是否有多个元素抢焦点？

#### 2. Alignment（对齐）
- 主要元素是否共享清晰边界？
- 表格、图表、图片、文本是否对齐？
- 页面之间是否有一致栅格？

#### 3. Spacing（留白）
- 组内元素是否靠近，组间是否分开？
- 页面是否拥挤？
- 边距是否稳定？

#### 4. Consistency（一致性）
- 字体、字号、颜色、图标、线条是否一致？
- 同类元素是否同样表现？
- 是否引入了无理由的新样式？

#### 5. Readability（可读性）
- 投屏/打印/移动端场景下是否可读？
- 对比度是否足够？
- 长文本是否换行良好？
- 图表标签是否清楚？

### Step 3：导出检查

必须看最终或接近最终的导出：
- PDF/DOCX/PPTX 是否能打开
- PNG/SVG 是否尺寸正确
- 字体是否缺失或替换
- 图片是否模糊、拉伸、裁切错误

### Step 4：记录审查证据

报告包含：
- 预览/导出路径
- 检查视口或页面范围
- 发现和严重性
- 是否批准交付

## Artifact Type 指南

| artifact_type | 审查重点 |
|---------------|---------|
| `deck` | 投屏可读、页面节奏、标题层级 |
| `document` | 阅读舒适、打印稳定、表格图注 |
| `visual` | 构图、规格、品牌一致性、导出质量 |

## 验证失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| 预览无法打开 | 阻塞，重新导出或修复源文件 |
| 字体缺失 | 替换为可用字体或嵌入字体 |
| 页面过载 | 删减、拆分、降低次要元素权重 |
| 对比度不足 | 调整颜色或字体权重后重新预览 |

## 常见说辞

| 说辞 | 现实 |
|------|------|
| "源文件看着没问题" | 交付的是导出物，不是你的编辑界面。 |
| "对齐差一点没人看得出来" | 对齐问题会让产物显得不专业，即使读者说不出原因。 |
| "文字小一点就放下了" | 牺牲可读性不是版式方案。 |
| "视觉稿不用验证" | 视觉稿的验证就是预览、尺寸、对比、导出质量。 |

## 红旗 — STOP

- 没有实际预览或导出物
- 多个主焦点互相竞争
- 文本溢出、遮挡、过小
- 图像比例失真或模糊
- 页面之间样式随机变化
- 导出规格不符合使用场景

## 验证清单

- [ ] artifact_type 已读取
- [ ] 实际预览或导出物已打开检查
- [ ] 层级、对齐、留白、一致性、可读性已审查
- [ ] 导出规格符合 spec
- [ ] 阻塞问题已列出
- [ ] 通过/退回结论明确
- [ ] 截图或预览路径已记录
